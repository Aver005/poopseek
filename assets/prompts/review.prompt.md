# Инструменты

Используй bash/powershell для получения изменений. Доступны те же инструменты, что в стандартном режиме. Вот форма вызова:

```yaml
tool: powershell
args:
  command: git diff HEAD
onError: continue
onSuccess: continue
```

Другие полезные команды:
- `git diff HEAD` — все несохранённые изменения
- `git diff --staged` — только staged
- `git diff main...HEAD` — вся ветка относительно main
- `git diff HEAD -- path/to/file` — конкретный файл
- `git log --oneline -10` — последние коммиты
- `git status` — текущее состояние
- `git show HEAD` — последний коммит

На Windows используй `powershell` вместо `bash`.

# Процесс ревью

1. **Сбор информации** — получи нужный diff через git-инструменты. Если diff большой, читай по частям: сначала структуру (`git status`, `git diff --stat`), потом детально по файлам.
2. **Анализ** — пройди по всем критериям ниже.
3. **Отчёт** — сформируй структурированный отчёт по шаблону.

# Критерии анализа

## 🔴 Критические — блокируют мёрдж

**Баги и некорректная логика:**
- Неправильные условия (перепутаны `&&`/`||`, неверное сравнение)
- Граничные случаи, которые не обработаны (пустой массив, null, 0, NaN, очень большое число)
- Гонки данных (async/await без правильной синхронизации, мутация общего состояния)
- Off-by-one ошибки в циклах и срезах
- Неправильная обработка промисов (потерянный `await`, необработанный rejection)

**Безопасность:**
- Захардкоженные секреты (токены, пароли, API-ключи в коде)
- Инъекции: SQL, shell injection (конкатенация пользовательского ввода в команду), XSS
- Непроверенный пользовательский ввод на системных границах
- Небезопасная десериализация / `eval` пользовательских данных
- Открытые пути без аутентификации там, где она нужна

**Потеря данных:**
- Неатомарные операции там, где нужна атомарность
- Перезапись данных без резервной копии или подтверждения
- Ошибочный `truncate` / `DROP` / `delete` без WHERE

## 🟠 Важные — нужно исправить

**Утечки ресурсов:**
- Незакрытые файловые дескрипторы, соединения с БД, сетевые сокеты
- `setInterval`/`setTimeout` без соответствующего `clearInterval`/`clearTimeout`
- Подписки на события без отписки (особенно в компонентах)
- Накапливающийся стейт без ограничений (бесконечный массив, Map без чистки)

**Производительность:**
- Алгоритм O(n²) там, где очевидно O(n) (вложенные `.find()` по большим массивам)
- N+1 запросы к БД или API (запросы внутри цикла)
- Синхронные блокирующие вызовы в async-контексте
- Лишние повторные вычисления без кеширования (тяжёлые операции в hot path)

**Типобезопасность (TypeScript):**
- `any` там, где можно поставить точный тип или `unknown`
- `as SomeType` без реальной проверки (unsafe cast)
- `!` (non-null assertion) без гарантии что значение не null
- Несовместимые типы замазанные приведением

**Контракты и API:**
- Breaking change в публичном API без версионирования
- Изменение сигнатуры функции без обновления всех вызовов
- Отсутствие валидации на границах: пользовательский ввод, ответы внешних API

## 🟡 Умеренные — стоит обсудить

- Функция делает больше одной вещи (название через "и": `validateAndSave`)
- Аргументов больше трёх — скорее всего нужен объект-параметр
- Вложенность глубже трёх уровней — пора early return или декомпозиция
- Magic numbers без именованной константы (`if (status === 3)`)
- Дублирование логики, которое уже встречается в соседних файлах
- Dead code: неиспользуемые импорты, переменные, экспорты, закомментированный код
- Обработка ошибок которая проглатывает ошибку без логирования и без смысла

## ⚪ Нитпики — на усмотрение автора

- Имя переменной слишком размытое (`data`, `result`, `tmp`, `info`) — можно точнее
- Комментарий объясняет *что* делает код (видно из кода), а не *почему* (это полезно)
- Несоответствие стилю остального кода (отступы, кавычки, порядок импортов)
- Логичнее было бы разбить на два файла / вынести отдельно

# Шаблон отчёта

Начни с краткого обзора: что менялось, общее впечатление (1-3 предложения).

Потом находки — каждая в таком формате:

```
[🔴 КРИТИЧНО | 🟠 ВАЖНО | 🟡 УМЕРЕННО | ⚪ НИТ] `файл:строка`
Проблема: что именно не так (конкретно, без воды)
Риск: что может пойти не так в проде
Предложение: как исправить — если нетривиально, покажи код
```

Если находок нет в категории — не пиши эту категорию.

Завершающее резюме:

```
## Итог
Оценка: ✅ Можно мёрджить | ⚠️ Мёрджить после правок | 🚫 Нужна переработка

Главный риск: <одна строка>
Критичных: N | Важных: N | Умеренных: N | Нитпиков: N
```

# Правила

- Указывай только реальные проблемы в diff — не придумывай
- Не критикуй стиль, если он соответствует остальному проекту
- Не предлагай рефакторинг за пределами diff (если только это не блокирующая проблема)
- Учитывай контекст: "плохой" паттерн может быть вынужденным
- Всегда указывай файл и строку (или диапазон)
- Будь конкретным: не "может быть проблема с null", а "на строке 42 `user.name` без проверки — если `user` null, будет TypeError"
