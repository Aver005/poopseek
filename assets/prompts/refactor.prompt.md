# Каталог стратегий

Каждая стратегия имеет: **триггеры** (что искать в проекте), **приоритет** (↑ высокий / → средний / ↓ низкий impact), **усилие** (★ мало / ★★ средне / ★★★ много), **плейбук** (конкретные шаги).

---

## S01 · Мёртвый код
**Триггеры:** неиспользуемые импорты, экспорты без вызовов, закомментированные блоки, zombie-функции
**Impact:** ↑ · **Усилие:** ★

**Плейбук:**
1. Найди неиспользуемые импорты: `grep -r "^import" src/ | ...` или TypeScript compiler errors с `noUnusedLocals`
2. Найди экспорты без единого импортера: `grep -rn "export" --include="*.ts" | ...`
3. Найди закомментированные блоки кода (не документации)
4. Удали найденное, убедись что сборка не сломана

---

## S02 · Магические значения → константы
**Триггеры:** числа и строки прямо в условиях (`=== 3`, `=== "admin"`, таймауты, лимиты), одинаковые литералы в нескольких местах
**Impact:** ↑ · **Усилие:** ★

**Плейбук:**
1. Найди повторяющиеся числовые и строковые литералы в логике
2. Создай `constants.ts` (или `config.ts`) в нужном модуле
3. Замени вхождения импортом константы
4. Особое внимание: магические числа в условиях, таймауты, HTTP-статусы, ключи хранилища

---

## S03 · Глубокая вложенность → ранний возврат
**Триггеры:** if/else-цепочки глубже 3 уровней, pyramid of doom, условия с несколькими `&&`/`||`, функции с отступом 4+ уровня
**Impact:** ↑ · **Усилие:** ★★

**Плейбук:**
1. Найди функции с глубокой вложенностью (`grep -n "^\s\{16,\}"`)
2. Применяй guard clauses: инвертируй условие и делай early return
3. Убери `else` после `return`
4. Длинные условия извлеки в именованные булевы переменные

---

## S04 · Укрепление типов TypeScript
**Триггеры:** `any` в аргументах или возвращаемых типах, `as SomeType` без реальной проверки, `!` assertion без гарантии, отсутствие дженериков там где они очевидны, `object` вместо конкретного типа
**Impact:** ↑ · **Усилие:** ★★★

**Плейбук:**
1. Найди все `any`: `grep -rn ": any\|as any\| any)" --include="*.ts"`
2. Для каждого `any` определи реальный тип или замени на `unknown` + type guard
3. Найди `as Type` — добавь runtime-проверку или перепиши без cast
4. Найди места где нужны дженерики (повторяющийся код с разными типами)
5. Добавь `strict: true` в tsconfig если ещё нет, разберись с ошибками

---

## S05 · Дублирование → переиспользование
**Триггеры:** одинаковые блоки логики в разных файлах, copy-paste с минимальными отличиями, похожие функции решающие одно и то же
**Impact:** ↑ · **Усилие:** ★★

**Плейбук:**
1. Найди дублирующиеся функции: читай файлы по очереди, запоминай паттерны
2. Выяви общую абстракцию (параметр вместо дублированного значения)
3. Создай shared-модуль (`utils/`, `shared/`, `helpers/`)
4. Замени все копии на импорт из общего места
5. Убедись что интерфейс общей функции покрывает все use-cases

---

## S06 · Имена → самодокументация
**Триггеры:** однобуквенные переменные (не в forEach/map), размытые имена (`data`, `result`, `handler`, `temp`, `info`, `obj`), функции с "и" в названии, несоответствие имени и поведения
**Impact:** → · **Усилие:** ★

**Плейбук:**
1. Scan all variable/function names in changed/key files
2. Переименуй: переменные → что хранит, функции → что делает, boolean → утверждение
3. Функции с "и" в имени — признак нарушения SRP, рассмотри разбиение
4. Используй find-and-replace по всему проекту (grep для поиска вхождений)

---

## S07 · Декомпозиция больших функций
**Триггеры:** функции длиннее 50 строк, несколько абзацев логики разделённых комментариями, функции делающие и вычисление, и I/O, и форматирование
**Impact:** ↑ · **Усилие:** ★★

**Плейбук:**
1. Найди длинные функции: `grep -n "^[[:space:]]*\(function\|const.*=.*=>\|async\)" src/ -A 60 | ...`
   Или читай файлы и выявляй визуально
2. Выдели логические блоки (часто отделены пустыми строками или комментариями)
3. Каждый блок → именованная вспомогательная функция
4. Убедись что новые имена точно описывают что делает блок

---

## S08 · Обработка ошибок
**Триггеры:** пустые catch-блоки (`catch {}`), `console.error` без действия, отсутствие try/catch на системных границах (I/O, сеть, парсинг), разные форматы ошибок в разных местах
**Impact:** ↑ · **Усилие:** ★★

**Плейбук:**
1. Найди все catch-блоки: `grep -n "catch" --include="*.ts" -r`
2. Пустые/заглушённые → добавь логирование или пробрось
3. Системные границы без обработки → оберни в try/catch
4. Создай единый тип ошибки если в проекте нет — все ошибки через него

---

## S09 · Async/Promise гигиена
**Триггеры:** floating promises (вызов async без await и без .catch), последовательные await которые можно параллелизовать, смешивание callbacks и promises, `.then().then()` вместо async/await
**Impact:** ↑ · **Усилие:** ★★

**Плейбук:**
1. Найди floating promises: вызовы async-функций без await, не присвоенные в переменную
2. Найди последовательные `await` без зависимостей → замени на `Promise.all`
3. Найди `.then()` цепочки → конвертируй в async/await
4. Проверь что все rejection обработаны (`.catch()` или try/catch)

---

## S10 · Модульные границы
**Триггеры:** файлы длиннее 300 строк, circular imports, смешанные уровни абстракции в одном файле, утечка внутренних деталей через публичные интерфейсы, index.ts экспортирует всё подряд
**Impact:** ↑ · **Усилие:** ★★★

**Плейбук:**
1. Найди крупные файлы: `find . -name "*.ts" | xargs wc -l | sort -rn | head -20`
2. Выяви circular imports (TypeScript покажет как ошибку или предупреждение)
3. Раздели god-файлы по ответственности (один файл = одна концепция)
4. Убедись что public API модуля минимален — прячь детали через не-экспортируемые функции
5. Проверь что `index.ts` экспортирует только то что реально нужно снаружи

---

## S11 · Конфигурация
**Триггеры:** хардкоженные URL, порты, имена хостов, пути к файлам разбросаны по коду; env-переменные читаются в разных местах; отсутствует валидация env при старте
**Impact:** → · **Усилие:** ★★

**Плейбук:**
1. Найди все обращения к `process.env` и хардкоженные конфиги
2. Создай центральный `config.ts` с валидацией (проверяет что нужные переменные заданы при старте)
3. Все остальные части кода импортируют из `config.ts`, не трогают env напрямую
4. Добавь типизированный объект конфига, не голые строки

---

## S12 · Согласованность API и интерфейсов
**Триггеры:** функции с похожей целью но разными сигнатурами, смешанные стили возврата (иногда `{ok, data}`, иногда голое значение, иногда throw), непоследовательное именование (camelCase vs snake_case)
**Impact:** → · **Усилие:** ★★

**Плейбук:**
1. Выяви паттерны: как функции возвращают результаты? как именуются параметры?
2. Определи доминирующий стиль (тот что используется чаще)
3. Приведи outliers к общему стандарту
4. Обнови все места вызова

# Уровни

## easy — Одна цель, максимальный impact
- Сканируй проект быстро (структура + ключевые файлы, 3-5 минут работы)
- Выбери **одну** стратегию с максимальным impact при минимальном усилии
- Выполни её полностью, без компромиссов
- Не трогай ничего за пределами выбранной стратегии
- Итого шагов: 1 стратегия · полное выполнение

## medium — Комплексное улучшение
- Сканируй проект тщательнее (читай ключевые файлы по архитектуре)
- Выбери **2–3 стратегии**, приоритизируй: сначала ★, потом ★★
- Выполняй одну за другой, проверяй сборку между стратегиями
- После каждой стратегии делай git-коммит
- Итого: 2–3 стратегии · сборка между ними · 2–3 коммита

## hard — Глубокий рефакторинг
- Прочти структуру проекта полностью, ключевые файлы — детально
- Выбери **4–6 стратегий**, отсортируй по порядку зависимостей
  (например: S01 мёртвый код → S04 типы → S10 модули)
- Выполняй методично, после каждой стратегии:
  - Проверь сборку
  - Сделай коммит с точным описанием (`refactor: eliminate dead code`)
- Используй `todo.write` для отслеживания
- Итого: 4–6 стратегий · 4–6 коммитов · полная валидация

## ultra — Полная трансформация
- Прочти ВЕСЬ проект: структуру, архитектуру, ключевые модули
- Оцени ВСЕ 12 стратегий
- Выбери все применимые (score ≥ 2), реши оптимальный порядок
- Выполняй как строительный проект: фундамент → структура → детали
  (мёртвый код и модули → типы → дублирование → остальное)
- Коммит после каждой стратегии
- Финальный отчёт: что изменилось, метрики (строки удалены, any устранены, etc.)
- Итого: все применимые стратегии · коммит каждая · финальный отчёт

# Алгоритм выбора стратегий

## Шаг 1 — Сканирование

Собери контекст:
```yaml
tool: powershell
args:
  command: |
    Write-Host "=== Структура ===" 
    Get-ChildItem -Recurse -Include "*.ts","*.tsx","*.js","*.jsx","*.py","*.go","*.rs" | 
      Where-Object { $_.FullName -notmatch "node_modules|build|dist|\.git" } | 
      Measure-Object | Select-Object -ExpandProperty Count
    Write-Host "=== Крупные файлы ==="
    Get-ChildItem -Recurse -Include "*.ts","*.tsx" | 
      Where-Object { $_.FullName -notmatch "node_modules|build|dist" } |
      ForEach-Object { [PSCustomObject]@{Lines=(Get-Content $_.FullName).Count; File=$_.FullName} } |
      Sort-Object Lines -Descending | Select-Object -First 10 | Format-Table -AutoSize
    Write-Host "=== any count ==="
    (Select-String -Path "src\**\*.ts" -Pattern ": any|as any" -Recurse).Count
    Write-Host "=== Catch blocks ==="
    (Select-String -Path "src\**\*.ts" -Pattern "catch" -Recurse).Count
onError: continue
onSuccess: continue
```

Дополнительно читай ключевые файлы проекта (index, main, крупнейшие модули).

## Шаг 2 — Скоринг

Для каждой стратегии проставь score 0–3:
- **3** — явные и многочисленные триггеры найдены
- **2** — триггеры есть, проблема реальная
- **1** — слабые признаки
- **0** — стратегия неприменима к данному проекту/стеку

Запиши скоры ПЕРЕД выбором — это дисциплинирует.

## Шаг 3 — Выбор

- **easy**: топ-1 по score. При равных — предпочти меньшее усилие (★)
- **medium**: топ-3 по score. Убери стратегии с усилием ★★★ если их score < 3
- **hard**: всё с score ≥ 2, максимум 6
- **ultra**: всё с score ≥ 1

Отсортируй выбранные в порядке выполнения:
1. Сначала S01 (мёртвый код) — чистая основа
2. Потом S10 (модули) если выбрана — структура
3. Потом S04 (типы), S05 (дублирование) — связанные изменения
4. Потом S03, S07, S08, S09 — детали реализации
5. В конце S02, S06, S11, S12 — косметика и согласованность

# Процесс выполнения

## Начало — план через todo.write
Сразу после выбора стратегий создай список задач:
```yaml
tool: todo.write
args:
  items:
    - id: "scan"
      content: Сканирование и выбор стратегий
      status: done
    - id: "s01"
      content: "S01 · Мёртвый код"
      status: in_progress
    - id: "s04"
      content: "S04 · Укрепление типов"
      status: pending
    - id: "commit"
      content: Финальный коммит и отчёт
      status: pending
onError: continue
onSuccess: continue
```

## Между стратегиями

Проверяй сборку (для medium/hard/ultra):
```yaml
tool: powershell
args:
  command: bun run build 2>&1 | tail -5
onError: continue
onSuccess: continue
```

Делай коммит (для medium/hard/ultra):
```yaml
tool: powershell
args:
  command: git add -A && git commit -m "refactor: <что именно сделано>"
onError: continue
onSuccess: continue
```

Обновляй todo после каждого шага.

## Конец — финальный отчёт

Подведи итог:
- Сколько файлов изменено
- Какие стратегии применены и что конкретно сделано
- Метрики: строки удалены, `any` устранено, дублей убрано — по возможности с числами
- Что осталось (намеренно не трогали)

# Дисциплина вызовов инструментов

Это самое важное правило для рефакторинга — нарушение приводит к порче файлов.

## Читай отдельно, пиши отдельно

**НЕЛЬЗЯ** смешивать чтение и запись в одном сообщении:
```
# Плохо — write вычислен до того как получен результат read:
[powershell: cat file.ts]  ← результат ещё не пришёл
[file.edit: ...]           ← пишет вслепую, на основе предположения
```

**ПРАВИЛЬНО** — два отдельных сообщения:
```
Сообщение 1: прочитать файл(ы) → получить результаты
Сообщение 2: записать изменения на основе реального содержимого
```

## Сколько инструментов за раз

- **Фаза чтения/разведки:** 3–5 инструментов за сообщение (несколько файлов сразу — окей)
- **Фаза записи:** 1–3 инструмента за сообщение. Написал → убедился → следующий
- **Никогда:** не вызывай более 5 инструментов записи подряд без промежуточной проверки

## Цикл работы с файлом

```
1. Прочитай файл (powershell/file.read) → жди результата
2. Проанализируй содержимое → запланируй изменения
3. Примени изменение (file.edit или powershell Set-Content) → жди результата
4. Убедись что ok. Если not ok — разберись с ошибкой прямо сейчас
5. Только потом переходи к следующему файлу
```

## Запись больших изменений

Для изменений > 10 строк используй `powershell` с `Set-Content`, а не `file.edit`:
```yaml
tool: powershell
args:
  command: |
    $content = @'
    // новое содержимое файла
    '@
    Set-Content -Path "src/foo.ts" -Value $content -Encoding UTF8
onError: continue
onSuccess: continue
```

`file.edit` для точечных замен строк. `powershell` для переписывания больших блоков.

# Правила рефакторинга

- **Не меняй поведение** — рефакторинг не должен ломать функциональность
- **Малые коммиты** — один коммит = одна стратегия
- **Проверяй сборку** перед коммитом (medium/hard/ultra)
- **Не рефакторь за пределами выбранных стратегий** — скоуп должен быть чётким
- **Если что-то сломалось** — откати именно этот шаг, не двигайся дальше
- **Именуй коммиты** точно: `refactor: eliminate dead code in auth module`
- **Для easy**: не трать время на полный скан — достаточно быстрой оценки
- **Для ultra**: не спеши — тщательный скан ценнее быстрого старта
